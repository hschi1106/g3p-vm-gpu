# PSB2 Support TODO

## Goal

Make the current `g3p-vm-gpu` architecture able to run all PSB2 benchmark testcases, then extend it so the system has a realistic chance to solve more of those tasks.

---

## Phase 1: Make PSB2 testcases ingestible and evaluatable

### 1. Extend the value system beyond scalar-only values

- [ ] Add `String` support to the core `Value` representation
- [ ] Add sequence support for lists
- [ ] Decide the internal representation:
  - Option A: `List[Value]`
  - Option B: typed lists such as `List[Int]` / `List[Float]`
- [ ] Keep the representation GPU-compatible
- [ ] Update both C++ and Python value encoders/decoders

Why:

- PSB2 datasets contain `str`, `list[int]`, `list[float]`, and empty lists
- current `Value` only supports `Int | Float | Bool | None`

### 2. Support multi-output tasks

- [ ] Replace single `expected: Value` fitness target with structured outputs
- [ ] Decide output representation:
  - tuple-like structure
  - list of outputs
  - named output map
- [ ] Update JSON fixture schema to store multiple outputs
- [ ] Update CPU fitness evaluation to compare multiple outputs
- [ ] Update GPU fitness evaluation to compare multiple outputs

Why:

- PSB2 includes tasks like `coin-sums`, `find-pair`, and `mastermind` with `output1`, `output2`, etc.

### 3. Add structured equality and fitness comparison

- [ ] Implement deep equality for strings
- [ ] Implement deep equality for lists
- [ ] Implement equality for multi-output results
- [ ] Define mismatch scoring for structured outputs
- [ ] Decide whether partial similarity should contribute to fitness

Why:

- current fitness logic assumes scalar outputs
- PSB2 outputs are often structured

### 4. Extend dataset ingestion and fixture conversion

- [ ] Add a converter from PSB2 examples into repo fixtures
- [ ] Preserve input order and output order
- [ ] Support empty sequences
- [ ] Support mixed scalar + sequence inputs
- [ ] Support multiple outputs in one case

Why:

- current `fitness-cases-v1` flow is built around scalar single-output cases

### 5. Redesign GPU data packing for variable-length values

- [ ] Replace scalar-only GPU `Value` handling with arena-based packing
- [ ] Add offset/length metadata for strings and lists
- [ ] Support host-to-device upload of variable-length inputs
- [ ] Support device-side read access for variable-length values
- [ ] Support copy-back / comparison for structured outputs

Why:

- current GPU path assumes compact fixed-size scalar values
- strings and lists cannot be represented safely with host pointers

## Recommended implementation order

1. Structured fixtures + multi-output support
2. `String` and `List` value representation
3. CPU-side deep equality and fitness for structured outputs
4. GPU-side arena packing for variable-length values
5. `len`, indexing, and dynamic loops
6. String/list builtins
7. Bytecode / VM / parity tests
8. PSB2 capability matrix and benchmark rollout

---

## Current gap summary

The main blockers are:

- scalar-only `Value`
- single-output fitness model
- no strings
- no lists
- no dynamic input-length loops
- no sequence operations
- GPU path assumes fixed-size compact values

So the first milestone is not "solve all PSB2 tasks". The first milestone is:

> make every PSB2 testcase representable, loadable, and comparable in the current evaluation pipeline.
