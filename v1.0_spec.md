# g3p-vm-gpu v1.0 實作規劃（Agent 可直接執行版）

## 0. 文件目的與完成定義

本文件定義 v1.0 的「可執行實作計劃」。

完成定義（DoD）：
1. 可吃 PSB2 全部 task 的原始 JSONL，轉成 `fitness-cases-v1`，並可批次執行。
2. 執行流程可穩定輸出每個 task 的演化歷史與摘要報告。
3. 擴充型別與語言後，CPU/GPU 結果在既定容差內一致。
4. 在標準基準上，維持接近目前等級的 GPU 加速比（見第 10 步）。
5. 演化有可觀測進步（best/mean fitness 隨世代改善；不要求一定解題成功）。

## 4. Value 型別擴充（String/List）

狀態：`已完成（v1.0 階段一：transport + equality）`（2026-03-05）

目前進度：
1. `ValueTag` 已新增 `String/List`。
2. CLI typed value 解碼已可讀取 `string/list`，並以 deterministic hash 形式封裝至 `Value`。
3. fitness equality 已支援 `String/List` tag 的一致性比較（hash 相等視為相等）。
4. PSB2 all-task 可執行題數已提升到 `25/25`。

尚未完成（留待第 5/6 步）：
1. String/List 完整 payload 與操作語義（目前仍為 hash transport）。
2. 語言層字串/陣列 builtin/opcode。

### 實作內容
1. 在 C++/GPU 可搬運的 `Value` 加入 String/List 表示方案。
2. 先決定資料布局（host/device 一致）：
- 固定上限 inline（小字串/小陣列）或
- arena/offset 模式
3. 更新 JSON codec（CLI 讀寫 typed value）。
4. 更新 equality/print/debug 序列化。

### 驗收標準
1. 新增型別後，舊數值流程行為不變（既有測試全綠）。
2. 新增型別最少 10 個測試（含 CPU/GPU 往返、序列化、比較）。
3. GPU 路徑可正確處理 String/List 的載入與回傳（至少 smoke test）。

## 5. 語言與 VM 操作擴充（字串/陣列）

狀態：`進行中（v1.0 階段一）`（2026-03-05）

目前進度：
1. 新增 builtin `len(x)`（`x` 為 `String/List`）於 CPU/GPU/Python VM 路徑。
2. `CALL_BUILTIN` id mapping 擴充 `4 -> len`（runtime extension）。
3. 新增 builtin `concat(a,b)`（`String+String` / `List+List`）於 CPU/GPU/Python VM 路徑。
4. `CALL_BUILTIN` id mapping 擴充 `5 -> concat`（runtime extension）。
5. 新增 builtin `slice(x,lo,hi)`（`String/List`）於 CPU/GPU/Python VM 路徑。
6. `CALL_BUILTIN` id mapping 擴充 `6 -> slice`（runtime extension）。
7. 新增 builtin `index(x,i)`（`String/List`）於 CPU/GPU/Python VM 路徑。
8. `CALL_BUILTIN` id mapping 擴充 `7 -> index`（runtime extension）。
9. AST/compiler/interpreter/evo generator 已新增 `CALL_LEN` + `CALL_CONCAT` + `CALL_SLICE` + `CALL_INDEX`，演化可直接生成。
10. VM 比較語義擴充：`String/List` 支援 `EQ/NE`，排序比較仍不支援。
11. 新增對應測試（C++ `test_vm_edges`/`test_evo_ast`、Python `test_eval`/`test_vm_equiv`/`test_cpp_vm_equiv`）並全綠。

### 實作內容
1. 在 spec 新增 v1.0 語義：
- 字串長度、拼接、索引、切片
- 陣列長度、拼接、索引、切片
2. 新增 builtin/opcode，並定義錯誤語義（TypeError/ValueError/IndexError）。
3. 同步更新：
- Python 參考解釋器
- C++ CPU VM
- C++ GPU VM

### 驗收標準
1. 每個新增操作都有：
- 正常案例
- 邊界案例
- 錯誤案例
2. interpreter vs CPU VM vs GPU VM 三方結果一致（既定容差內）。
3. 不允許新增操作讓 GPU 專用語義偏離 spec。

## 6. 編譯器與 kernel 同步支援新型別

### 實作內容
1. 擴充 AST/bytecode/compiler，讓 String/List 相關節點可編譯。
2. 擴充 GPU kernel 執行路徑，確保 opcode coverage 完整。
3. 建立 opcode 覆蓋檢查（至少統計每個 opcode 是否被測到）。

### 驗收標準
1. 新型別 opcode 在 CPU/GPU 皆可執行，不出現「CPU 能跑、GPU 不支援」。
2. 新增一組跨後端 parity 測試（至少 50 個 mixed programs）。
3. 若 GPU 不支援某語義，必須在 compile 階段顯式拒絕，不能 runtime silent wrong。

## 7. fitness 比對語義擴充（容器型）

### 實作內容
1. 定義 `values_equal_for_fitness_v1`：
- scalar（int/float/bool/none）
- string
- list（遞迴比較，含長度與元素）
2. 定義 numeric 容差策略（float abs/rel）。
3. 定義 mismatch penalty 與 runtime error penalty 對容器型的規則。

### 驗收標準
1. CPU 與 GPU 的 fitness 計算誤差 <= `1e-9`（double score）。
2. 針對容器型輸出至少 20 個 parity 測試。
3. 對 NaN/Inf、深層巢狀 list、型別不匹配有明確可重現行為。

## 12. 風險與回退策略

1. 風險：String/List 在 GPU 記憶體布局過於複雜
- 回退：先支援 bounded container（長度上限），超出直接 compile-time reject。
2. 風險：all-task 某些題目語義差異太大
- 回退：標記 unsupported 並保留錯誤碼，不阻塞整體進度。
3. 風險：加速比下降
- 回退：保留 fast path（debug validate 預設關閉）、用 profile 找 kernel/傳輸瓶頸再優化。
